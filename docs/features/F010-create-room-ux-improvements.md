# F010: CreateRoomModal 信息层级与交互效率优化

> Status: done | Owner: 宪宪 | Completed: 2026-04-16

## Why

当前 `CreateRoomModal` 的 UX 问题不在视觉（F009 已统一），而在**信息层级、选择效率、提交闭环**三块。

铲屎官实测反馈：主任务"选人并发起讨论"的路径被三个因素干扰：
1. **工作目录**是高级选项，放在专家选择之前，争夺注意力
2. **筛选只能靠标签**，专家数量上来后查找成本快速恶化
3. **主 CTA 不固定**，用户滚到中段后看不到提交动作和当前选择结果

## What

### P0 — 必须修

| # | 问题 | 根因 | 修复方向 |
|---|------|------|----------|
| P0-1 | 工作目录抢占主任务注意力 | 工作目录 section 在专家选择之前，视觉权重不低 | 移至底部折叠区域，默认收起 |
| P0-2 | 专家筛选只有标签，没有搜索 | `activeTag` 只能按标签筛选 | 增加文本搜索框，标签筛选降为辅助 |
| P0-3 | 主 CTA 在长列表下不可达 | 底部按钮在小屏/专家多时滚出视口 | CTA sticky 固定在视口底部，始终可见 |
| P0-4 | 主按钮文案"加入讨论"术语不准确 | 当前动作是创建 Room，不是加入已有讨论 | 改为"创建讨论" |
| P0-5 | 错误只在底部一行红字，无字段绑定 | `error` state 只渲染在底部，不指向具体字段 | 每个字段旁内联错误提示，失败时自动聚焦到问题字段 |

### P1 — 应该修

| # | 问题 | 根因 | 修复方向 |
|---|------|------|----------|
| P1-1 | 讨论主题在视觉上权重接近其他字段 | 主题输入框与目录、筛选、已选区视觉权重相近 | 主题输入提升视觉层级，拉开与下层字段的间距 |
| P1-2 | 主持人选择没有空状态解释 | `managers` 为空时无说明；manager 多时缺少判断辅助 | 空状态加说明文字；多 manager 时显示 roleLabel |
| P1-3 | 筛选后无结果无直接恢复操作 | 只说"该领域暂无专家"，无"清除筛选" | 无结果时显示"清除筛选"操作入口 |
| P1-4 | 已选专家区反馈不够强 | "已选专家"区视觉上是结果展示，不是强反馈面板 | 改为"配置摘要"区，实时显示提交就绪状态 |

## In Scope

- `CreateRoomModal.tsx` 的表单结构重组
- 主题输入的视觉层级提升
- 主持人选择的空状态和判断辅助
- 专家网格的搜索框 + sticky CTA
- 工作目录移入折叠区域（默认收起）
- 字段级错误内联 + 失败时聚焦
- "配置摘要"替代"已选专家"区

## Out of Scope

- API 改动（`/api/rooms` 接口不变）
- 主持人功能逻辑（F004 路由器已有主持人机制）
- 目录选择器组件逻辑（F006 已有 DirectoryPicker）
- 邀请专家的拖拽/批量操作

- [x] AC-3: 主 CTA 按钮在滚动时 sticky 固定在视口底部，始终可达
- [x] AC-4: 主按钮文案改为"创建讨论"，术语准确
