# F051: 单端口统一入口网关

> **Status**: in-progress | **Owner**: codex | **Priority**: P1

## Why

当前本地运行默认同时暴露两个端口：

- 前端 UI：`7002`
- 后端 API / Socket：`7001`

这套结构对实现方来说没问题，但对使用者不友好。用户当前的核心诉求很明确：

> “能不能一键启动一个代理服务器，用户直接走一个端口就好了，剩下的代理服务器来路由，这样用户其实关注的也少，只要关注一个端口就好了。”

这个 Feature 的目标不是把所有环境都强行改成同一启动方式，而是明确区分两类场景：

- **开发态**：允许继续前后端分开运行，优先保证开发效率
- **生产态**：必须通过统一入口对外提供服务，用户只需要记住一个地址

也就是说，真正要收敛的是**用户访问体验**，不是开发者的内部工作流。

## What

### Phase A: 生产统一入口网关

生产运行时增加统一网关进程，对外只暴露 `7000` 作为用户入口，内部继续保留现有服务端口，并按路径转发：

- 页面请求转发到前端进程
- `/api/*` 转发到后端进程
- `/socket.io/*` 或等价实时通道转发到后端进程
- 前端内部默认仍为 `7002`
- 后端内部默认仍为 `7001`

生产用户视角只需要访问 `7000`；前后端是否仍是两个内部进程，对用户透明。

### Phase B: 开发兼容与文档收口

补齐开发/生产双模式下的配套：

- README 与启动日志明确区分生产入口 `7000` 和开发入口 `7001/7002`
- 前端默认 API 基址与实时连接策略改为兼容“生产走网关、开发可直连”两种模式
- 保留开发者绕过网关直连前后端的能力，作为明确支持的开发路径
- 明确 health / debug / API 文档在生产和开发下的访问方式

## Acceptance Criteria

### Phase A（生产统一入口网关）
- [ ] AC-A1: 生产运行时，对外统一入口固定为 `http://localhost:7000`
- [ ] AC-A2: `7000` 同时支持页面、REST API 和实时流量转发
- [ ] AC-A3: 网关背后继续保留前端 `7002`、后端 `7001` 作为内部默认端口
- [ ] AC-A4: 代理层故障时能输出明确错误，而不是静默白屏或无限挂起

### Phase B（开发兼容与文档收口）
- [ ] AC-B1: 开发运行时，允许继续以前端 `7002` + 后端 `7001` 的分离方式工作
- [ ] AC-B2: README、启动日志和开发说明都明确区分生产入口 `7000` 与开发入口 `7001/7002`
- [ ] AC-B3: 直接运行前端/后端子项目时，仍有清晰的开发者路径，不会因为生产默认改成网关而失效

## Risk

| 风险 | 缓解 |
|------|------|
| 只代理 HTTP，遗漏 Socket / 升级链路，导致房间实时更新失效 | AC 明确要求页面、API、实时流一起验收，设计上把 `/socket.io` 作为一等入口 |
| 生产入口新增 `7000`，但前端/后端默认端口约束仍需保留，容易在实现时被误改 | 明确 `7000` 仅为生产统一入口，内部默认端口继续固定为 `7001/7002` |
| 文档仍混淆“开发分离模式”和“生产统一入口”，导致使用者困惑 | 把 README、启动日志、运行说明一起纳入 Phase B，并明确区分场景 |

## Key Decisions

| # | 决策 | 理由 | 日期 |
|---|------|------|------|
| KD-1 | 该需求按 Feature 立项，而不是零散脚本修改 | 影响启动方式、文档、网络入口和实时链路，属于用户可感知能力变化 | 2026-04-23 |
| KD-2 | 生产统一入口使用 `7000` | 用户侧只关注一个端口，且不破坏内部 `7001/7002` 约束 | 2026-04-23 |
| KD-3 | 开发态继续允许前后端分离运行 | 开发效率优先，不把生产部署约束强加到本地开发工作流 | 2026-04-23 |
| KD-4 | 生产网关采用独立 Node 进程 | Socket.IO 与统一入口路由更直接，避免把网关职责耦合进 Next | 2026-04-23 |
| KD-5 | 提供可选 `dev:gateway` | 本地可以模拟生产单入口，但不强迫所有开发者都走网关 | 2026-04-23 |
